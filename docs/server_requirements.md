## Lattice server requirements (serial + mesh)

This document describes everything the server must implement to communicate with the master node over USB serial and control/query all mesh nodes. It includes the wire protocol, message schema, control opcodes, and reference implementation notes (with Go in mind).

### Topology overview
- One ESP32 node (the master) is connected to the server over USB serial.
- The master participates in the ESP-NOW mesh. The server does NOT speak ESP-NOW; it talks only to the master over serial.
- The firmware bridges between serial and mesh using a compact framed Protobuf.

### Transport
- Serial settings: 115200 baud, 8N1.
- Framing: 2-byte little-endian length prefix followed by a Protobuf-encoded payload (see schema below).
  - length = number of bytes in the Protobuf message that follows.
  - Multiple frames are sent back-to-back; the server must respect framing boundaries.

### Protobuf schema
Use this schema on the server to (de)serialize payloads. The server should generate code (e.g., with protoc) and use it to encode/decode frames.

```proto
syntax = "proto3";
package mesh;

// Set go_package, java_package, etc. as appropriate for your build

message MeshMessage {
  // Message type as seen/used on the wire
  // 0 = ADAPTER_DATA (normal adapter-originated data)
  // 1 = MASTER_BEACON (mesh-internal heartbeat from master)
  // 3 = SERIAL_CMD_BROADCAST (special: server→device serial command to broadcast adapter data)
  uint32 messageType = 1;

  // Adapter data type (maps to firmware enum adapter_types)
  //   -1 UNKNOWN
  //    0 PIR
  //    1 WIFI (reserved)
  //    2 LED  (reserved)
  //    3 SERIAL (serial control / health / commands)
  sint32 dataType = 2; // zigzag encoded

  // MAC addresses (6 bytes used). The server only needs to send targetMacAddress.
  // originMacAddress, lastHopMacAddress, and hopCount are automatically generated by the device.
  bytes originMacAddress = 3;   // 6 bytes (auto-generated by device)
  bytes targetMacAddress = 4;   // 6 bytes (required from server)
  bytes lastHopMacAddress = 5;  // 6 bytes (auto-generated by device)

  // 12-byte adapter payload. For SERIAL (dataType=3) this contains control opcodes.
  bytes data = 6; // up to 12 bytes used

  // Mesh hop count (0 for locally originated)
  uint32 hopCount = 7; // auto-generated by device
}
```

Notes:
- For normal mesh traffic (device→server) `messageType` will be 0 (ADAPTER_DATA) or 1 (MASTER_BEACON).
- For server→device broadcast commands, set `messageType` to 3 (SERIAL_CMD_BROADCAST). The device will convert this into a mesh broadcast on your behalf.
- **Important**: When sending messages from server to device, you only need to populate:
  - `messageType` (required)
  - `dataType` (required) 
  - `targetMacAddress` (required for targeted messages)
  - `data` (required)
  - The device automatically generates: `originMacAddress`, `lastHopMacAddress`, and `hopCount`

### Adapter type enum (firmware values)
- UNKNOWN = -1
- PIR = 0
- WIFI = 1 (reserved)
- LED = 2 (reserved)
- SERIAL = 3

### Serial control opcodes (only when dataType = SERIAL)
Opcodes live in `MeshMessage.data[0]` and drive configuration/health.

- OP_CONFIG_SET (0xA0): Set adapter type on one node or all nodes
  - Payload layout in `data` (12 bytes total):
    - Byte 0: 0xA0
    - Bytes 1..6: target MAC address
      - Set to FF:FF:FF:FF:FF:FF to broadcast to all nodes
    - Byte 7: adapter type (int8, see enum above)
    - Bytes 8..11: reserved (0x00)
  - Behavior:
    - The target node writes the adapter type to EEPROM.
    - The target node reboots. On boot, the new adapter is constructed using the correct pin for that adapter type (automatically inferred).

- OP_HEALTH_REQ (0xB0): Request health reports
  - Payload layout:
    - Byte 0: 0xB0
    - Remaining bytes: 0x00
  - Behavior:
    - Any node receiving this (routed through mesh) will emit a health report immediately (see below). Nodes also report periodically.

- OP_HEALTH_REPORT (0xB1): Node → server health status
  - Emitted automatically by each node every ~5 seconds and on OP_HEALTH_REQ
  - Payload layout in `data`:
    - Byte 0: 0xB1
    - Byte 1: adapter type (current, int8)
    - Bytes 2..7: MAC address (6 bytes)
    - Bytes 8..11: uptime seconds (uint32 LE)

### Server responsibilities
1) Serial framing
   - Read 2 bytes (little-endian length), then read that many bytes for the Protobuf payload. Repeat.
   - When writing, marshal the Protobuf then prefix with 2-byte little-endian length.

2) Simplified message creation
   - **Outgoing messages (server → device)**: Only populate the essential fields:
     - `messageType`: The type of message (0=ADAPTER_DATA, 3=SERIAL_CMD_BROADCAST)
     - `dataType`: The adapter type for the data
     - `targetMacAddress`: For targeted messages (can be broadcast FF:FF:FF:FF:FF:FF)
     - `data`: The actual payload (up to 12 bytes)
   - **Auto-generated fields**: The device automatically sets:
     - `originMacAddress`: Set to the device's own MAC address
     - `lastHopMacAddress`: Set to the device's own MAC address (since it's the origin)
     - `hopCount`: Set to 0 (since it's locally originated)

3) Protobuf encoding/decoding
   - Generate code for the schema and use it to parse incoming frames and build outgoing frames.

4) Command routing (server → master → mesh)
   - To send adapter data to the master (which routes toward the mesh master path):
     - Set `messageType = 0` (ADAPTER_DATA)
     - Set `dataType` appropriately
     - Fill `data` (<= 12 bytes)
     - Write frame over serial
   - To broadcast adapter data to all nodes:
     - Set `messageType = 3` (SERIAL_CMD_BROADCAST)
     - Set `dataType` and `data` as above
   - To set adapter type on a node:
     - Build message with `dataType = SERIAL`, `messageType = 0`
     - Put OP_CONFIG_SET payload into `data` (see opcode spec)
     - The mesh routes it; target node reboots and comes back with the new adapter (and pin) from EEPROM.
   - To request health:
     - Build message with `dataType = SERIAL`, `messageType = 0`, `data[0] = 0xB0`

5) State tracking (server)
   - Maintain a map of node MAC → last-seen health report:
     - current adapter type
     - uptime, last seen timestamp
   - Optionally, track route info using `hopCount` and MAC fields for diagnostics.

### EEPROM persistence
- EEPROM addresses (internal to firmware):
  - Adapter type: address 8 (int8)
- On first boot (unset EEPROM), firmware sets PIR as default adapter.
- After OP_CONFIG_SET, the node reboots to cleanly switch adapters.

### Go reference (sketch)
Pseudocode showing the key parts (uses go.bug.st/serial and google.golang.org/protobuf):

```go
// open serial
port, _ := serial.Open("COM3", &serial.Mode{BaudRate: 115200})
defer port.Close()

// write frame
func writeFrame(p serial.Port, m *mesh.MeshMessage) error {
  b, _ := proto.Marshal(m)
  hdr := []byte{byte(len(b)), byte(len(b) >> 8)}
  if _, err := p.Write(hdr); err != nil { return err }
  _, err := p.Write(b)
  return err
}

// read frame
func readFrame(p serial.Port) (*mesh.MeshMessage, error) {
  hdr := make([]byte, 2)
  if _, err := io.ReadFull(p, hdr); err != nil { return nil, err }
  n := int(hdr[0]) | int(hdr[1])<<8
  buf := make([]byte, n)
  if _, err := io.ReadFull(p, buf); err != nil { return nil, err }
  var msg mesh.MeshMessage
  if err := proto.Unmarshal(buf, &msg); err != nil { return nil, err }
  return &msg, nil
}

// set adapter type on a node
func sendConfigSet(targetMac [6]byte, adapterType int8) error {
  payload := make([]byte, 12)
  payload[0] = 0xA0 // OP_CONFIG_SET
  copy(payload[1:7], targetMac[:])
  payload[7] = byte(adapterType)
  // Bytes 8-11 are reserved (0x00) - pin is automatically inferred from adapter type
  
  m := &mesh.MeshMessage{
    MessageType:     0, // ADAPTER_DATA
    DataType:        3, // SERIAL
    TargetMacAddress: targetMac[:], // Only required field for routing
    Data:            payload,
    // Note: originMacAddress, lastHopMacAddress, and hopCount are auto-generated by the device
  }
  return writeFrame(port, m)
}

// broadcast adapter data
func broadcastAdapterData(dataType int32, data []byte) error {
  m := &mesh.MeshMessage{ 
    MessageType: 3, // SERIAL_CMD_BROADCAST
    DataType:    dataType, 
    Data:        data,
    // Note: all MAC addresses and hopCount are auto-generated by the device
  }
  return writeFrame(port, m)
}

// consume health reports
for {
  msg, err := readFrame(port)
  if err != nil { continue }
  if msg.DataType == 3 && len(msg.Data) >= 12 && msg.Data[0] == 0xB1 { // OP_HEALTH_REPORT
    nodeType := int8(msg.Data[1])
    mac := msg.Data[2:8]
    uptime := uint32(msg.Data[8]) | uint32(msg.Data[9])<<8 | uint32(msg.Data[10])<<16 | uint32(msg.Data[11])<<24
    // update your node registry
    _ = nodeType; _ = mac; _ = uptime
  }
}
```

### Message flow examples
- Configure adapter on a specific node:
  1) Server → master (serial): `dataType=SERIAL`, `data=[A0][targetMac][type]`
  2) Master relays through mesh
  3) Target node saves type to EEPROM and reboots
  4) Shortly after reboot, node resumes sending periodic health (0xB1)

- Broadcast health request:
  1) Server → master: `dataType=SERIAL`, `data=[B0]`
  2) Nodes emit 0xB1 reports immediately (and periodically)

- Broadcast an adapter payload (e.g., app-specific):
  1) Server → master: `messageType=3`, `dataType=<your adapter>`, `data[0..]`
  2) Master broadcasts to all peers

### Operational guidance
- The server should treat the master as the single I/O tunnel. It does not need to know the mesh topology.
- Persist any desired configuration state in your server DB keyed by node MAC; use periodic health to reconcile.
- If a node is not reporting health for a while, it is out of range or offline. Use lastSeen timestamps client-side.

### Error handling
- The device may drop frames with invalid length or malformed Protobufs silently.
- If you send an unsupported adapter type, the node will store it but the creation may fail at boot; watch for missing health reports or treat as misconfiguration.

### Security
- This protocol is not authenticated. For production, consider adding an application-level signature/MAC inside the 12-byte `data` or switching to an encrypted host link.

### 2025-02 Encryption & Beacon notes

* All ESP-NOW frames are **AES-encrypted**.  The LMK is the 16-byte `DEFAULT_MESH_KEY` in `project_config.h` and is identical on every node.  The server must either know this key or treat frames as opaque binary.
* Every master beacon is now sent **twice**: once as raw broadcast (ESP-NOW addr=`FF:FF:FF:FF:FF:FF`) and once unicast to each peer.  Gateways should ignore the broadcast copy when proxying traffic to the cloud to avoid duplicates.

### Enrollment Protocol (node provisioning)

When a node boots for the first time (or after an EEPROM wipe) it is
**unenrolled**. An unenrolled node:

1. Prints its Curve25519 public key to serial (master node only relays):
   ```
   LATTICE_PUBKEY:3a7f2b...  (64 hex chars = 32 bytes)
   ```
2. Broadcasts `MESH_TYPE_ENROLLMENT` (type=2) messages every 10 seconds with
   `enrollmentPublicKey` populated.
3. Refuses to forward sensor data until it receives a `MESH_TYPE_JOIN_ACK`.

**Server responsibilities for enrollment:**

1. **Detect enrollment frames** — watch for incoming frames with `messageType == 2`.
   Extract the 32-byte `enrollmentPublicKey` and the `originMacAddress`.

2. **Approve the node** — your server decides whether to admit the node (user
   confirmation, allowlist check, etc.).

3. **Send JOIN_ACK** — construct a `MeshMessage` and write it to serial:
   ```go
   m := &mesh.MeshMessage{
     MessageType:      4,               // MESH_TYPE_JOIN_ACK
     DataType:         3,               // SERIAL_ADAPTER
     TargetMacAddress: nodeMac[:],      // 6-byte MAC of the unenrolled node
     Data:             make([]byte, 12), // all zeros; content ignored
   }
   writeFrame(port, m)
   ```
   The master relays this to the target node over the mesh.

4. **Node becomes enrolled** — the node sets its enrolled flag in EEPROM and
   begins forwarding sensor data. It will no longer broadcast enrollment frames.

**Key points:**
- The server should persist approved public keys (keyed by MAC address) for
  future session continuity checks.
- If the serial framing drops the JOIN_ACK the node will retry every 10 seconds.
- `enrollmentPublicKey` is zero-filled (`\x00` × 32) in all non-enrollment messages.

#### Enrollment message flow

```
[node boot]
    Node → (ESP-NOW broadcast) → Master: MESH_TYPE_ENROLLMENT (pubkey inside)
    Master → (serial frame) → Server: forwards enrollment frame

[server approves]
    Server → (serial frame) → Master: MESH_TYPE_JOIN_ACK (targetMAC = node MAC)
    Master → (ESP-NOW unicast) → Node: JOIN_ACK

[node enrolled]
    Node → begins sending MESH_TYPE_ADAPTER_DATA
```
