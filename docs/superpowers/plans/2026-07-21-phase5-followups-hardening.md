# Phase 5: Follow-ups + Hardening Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Clear the tracked follow-ups accumulated across Phases 1–4: properly regenerate the nanopb serial codec (removing the fragile hand-edit), honor downlink commands from the secondary master post-failover, stop a relay from corrupting an AAD-bound field on sealed frames, and close small robustness + test-coverage gaps.

**Architecture:** Mostly independent fixes. The one cross-repo item (nanopb `max_size` options) lands in `lattice-protocol` first, is released as a tag, then consumed by a submodule bump + regen here — same release discipline as Phase 1. The rest are `lattice-nodes`-only.

**Tech Stack:** C++ (ESP32 firmware, host-tested), GoogleTest + `tests/e2e` sim harness. nanopb 0.4.9.1 generator (pip: `~/Library/Python/3.9/lib/python/site-packages/nanopb/generator/nanopb_generator.py` — confirmed exact version match with the committed header). `main/lib/lattice-protocol` submodule currently v0.4.0.

## Global Constraints

- Follow-ups sourced from the Phase 1–4 whole-branch reviews (see `.superpowers/sdd/progress.md`) and memory `nanopb-serial-schema-fragility`.
- **Explicitly DEFERRED (user decision):** the `currentMaster.distance` sticky-low minor is NOT fixed — the current sticky-min correctly picks the shortest path in the common multi-path case, it self-recovers within `STALE_MASTER_THRESHOLD_MS` (~9s), and the naive fix risks route oscillation. Document it as accepted; do not implement.
- **No wire-format change to the ESP-NOW `mesh_message`** (`lattice-protocol/c/mesh_message.h`, `sizeof==242`) — unchanged. This phase only touches the nanopb SERIAL codec (`mesh.pb.*` + `SerialFraming`) and firmware logic.
- **Release discipline (memory `lattice-protocol-release-flow`):** the `lattice-protocol` `mesh.options` change must be merged AND tagged (next in the `v0.x.0`/`v0.x.y` series after v0.4.0) before the nodes submodule re-points at the tag. No floating-branch-SHA submodule bump.
- **nanopb regen must reproduce the current serial struct shape for tags 15/16**: `secondaryMasterMac`/`secondaryPublicKey` must stay `optional bytes` with `max_size` (→ `PB_BYTES_ARRAY_T` with `has_`/`size`/`bytes`), NOT `fixed_length` (which would drop the `has_` flag and break `SerialFraming.cpp`'s `.has_`/`.size`/`.bytes` access). Only `max_size` (no `fixed_length`) for 12–16, matching the existing `public_key` (max_size:32) precedent.
- **CONFIG_SET-from-secondary security:** the real authentication gate is the E2E open (`nodeOpened`, k_down) — a forged CONFIG_SET cannot be opened without the key. Widening the *origin* check to also accept the secondary master's MAC is defense-in-depth relaxation only, and is safe because the leaf legitimately holds the secondary's key. Do NOT touch the `nodeOpened`/`needsOpen` gate (Phase-3 command-injection fix); only widen the origin comparison.
- All firmware test hooks `#ifdef UNIT_TEST`-gated.
- **clang-format 18** (CI `lint-format` checks `main/src` only, excludes `mesh.pb.h`/`mesh.pb.c` + `nanopb/`). `CF=$(python3 -c "import clang_format,os;print(os.path.join(os.path.dirname(clang_format.__file__),'data','bin','clang-format'))")`; check changed non-generated `main/src` files. Only reformat changed lines.
- Verification: `cmake -B tests/build tests/ -DCMAKE_BUILD_TYPE=Release` (once), `cmake --build tests/build --parallel`, `ctest --test-dir tests/build --output-on-failure`. Baseline **180 pass / 0 disabled**.
