# Message Route Tracking & Protocol Migration — Design

**Date:** 2026-07-13
**Status:** Approved
**Repos:** lattice-protocol (primary), lattice-nodes (firmware)

## Goal

1. Track the full MAC hop chain a message traverses through the mesh, enabling exact topology inference in the artist portal mesh map (replacing zone-inference heuristic).
2. Move `MeshMessageType`, `mesh_message` struct, and `mesh.proto` into `lattice-protocol` as the single source of truth, generated from Go source.

## Approach

Dedicated `MESH_TYPE_ROUTE_REPORT` message sent every 60s (2× health interval) from each non-master node. As the message relays upward, each intermediate node appends its own MAC to the `data` payload before forwarding. Server receives the complete hop chain in one atomic message.

Functional messages (`MESH_TYPE_ADAPTER_DATA`, health reports, etc.) are unchanged — no route overhead on the hot path.

## Architecture

### lattice-protocol

Single Go source of truth. `go generate ./...` produces three artefacts:

| Go source | Generated output |
|---|---|
| `message/types.go` | `c/message_types.h` |
| `message/message.go` | `c/mesh_message.h`, `proto/mesh.proto` |
| `opcodes/opcodes.go` | `c/opcodes.h` (existing) |
| `adapter/types.go` | `c/adapter_types.h` (existing) |

`cmd/gen-headers` is extended to emit `.proto` files in addition to C headers.

#### `message/types.go`

```go
const (
    MeshTypeAdapterData        = uint8(0)
    MeshTypeMasterBeacon       = uint8(1)
    MeshTypeEnrollment         = uint8(2)
    MeshTypeSerialCmdBroadcast = uint8(3)
    MeshTypeJoinAck            = uint8(4)
    MeshTypeRouteReport        = uint8(5) // dedicated route topology report
)
```

#### `message/message.go`

```go
// MeshMessage is the wire-format message struct for the Lattice mesh.
// Struct tags drive codegen: `c` → packed C struct field, `proto` → proto3 field number + type.
type MeshMessage struct {
    ProtoVersion        uint8    `c:"uint8_t"     proto:"10,uint32"`
    MessageType         uint8    `c:"uint8_t"     proto:"1,uint32"`
    DataType            int32    `c:"int32_t"     proto:"2,sint32"`
    OriginMacAddress    [6]byte  `c:"uint8_t[6]"  proto:"3,bytes"`
    TargetMacAddress    [6]byte  `c:"uint8_t[6]"  proto:"4,bytes"`
    LastHopMacAddress   [6]byte  `c:"uint8_t[6]"  proto:"5,bytes"`
    Data                [64]byte `c:"uint8_t[64]" proto:"6,bytes,optional"`
    HopCount            uint8    `c:"uint8_t"     proto:"7,uint32"`
    EpochNum            uint32   `c:"uint32_t"    proto:"8,uint32"`
    SeqNum              uint16   `c:"uint16_t"    proto:"9,uint32"`
    EnrollmentPublicKey [32]byte `c:"uint8_t[32]" proto:"11,bytes,optional"`
}
// Wire size: 127 bytes packed. Static assert emitted into c/mesh_message.h.
```

#### New opcode in `opcodes/opcodes.go`

```go
OpRouteReport = byte(0xB3) // Node→server: routing path; payload: [B3][1B path_len][path_len × 6B MACs]
```

### lattice-nodes firmware

#### Header migration

`Mesh.h` removes the inline `MeshMessageType` enum and `mesh_message` struct. Replaces with:

```cpp
#include "../../lib/lattice-protocol/c/message_types.h"
#include "../../lib/lattice-protocol/c/mesh_message.h"
```

`lib/lattice-protocol` is the existing submodule — no new dependency.

`mesh.pb.h` continues to be generated from `proto/mesh.proto` via nanopb. Source of truth moves to `lattice-protocol/proto/mesh.proto`; the copy in `main/proto/mesh.proto` is replaced by the generated file from the submodule.

#### New config constant (`project_config.h`)

```cpp
constexpr uint32_t ROUTE_REPORT_INTERVAL_MS = HEALTH_REPORT_INTERVAL_MS * 2; // 60 000 ms
```

#### Route report timer (`Mesh` private)

```cpp
uint32_t lastRouteReportMs;
```

Initialised to `0` in constructor. Checked in `loop()`:

```cpp
if (!isMaster &&
    millis() - lastRouteReportMs >= lattice::config::ROUTE_REPORT_INTERVAL_MS) {
    sendRouteReport();
    lastRouteReportMs = millis();
}
```

#### `sendRouteReport()`

Builds a `MESH_TYPE_ROUTE_REPORT` message with:

```
data[0] = OpRouteReport   (0xB3)
data[1] = 0               path_len — incremented by each relay
data[2..63] = zeroed      relay MAC slots, filled in-transit
```

Transmitted via `transmitCore(ADAPTER_TYPE_UNKNOWN, data, MESH_TYPE_ROUTE_REPORT)`.

#### Receive dispatch

New case added to the message type switch in `drainRecvQueue` / dispatch logic:

```cpp
case MESH_TYPE_ROUTE_REPORT:
    processRouteReport(msg);
    break;
```

#### `processRouteReport()`

Behaviour varies by node role:

**Serial (master-adjacent) node:** passes message up to server unchanged — the same path as `MESH_TYPE_ADAPTER_DATA` uplink.

**Relay (non-serial) node:**

1. Verify `data[0] == OpRouteReport` — drop if malformed.
2. Bounds check: `data[1] < 10` — drop if path full (10 × 6 = 60 bytes = data[2..61]).
3. Write own MAC: `memcpy(&data[2 + data[1] * 6], deviceMacAddress, 6)`.
4. Increment: `data[1]++`.
5. Update `lastHopMacAddress = deviceMacAddress`, `hopCount++`.
6. Forward toward master via `transmitCore`.

**Path budget:**

| Byte | Content |
|---|---|
| `data[0]` | `0xB3` OpRouteReport |
| `data[1]` | path_len (0–10) |
| `data[2..7]` | relay 1 MAC |
| `data[8..13]` | relay 2 MAC |
| … | … |
| `data[56..61]` | relay 10 MAC |
| `data[62..63]` | unused (2 bytes spare) |

**Overflow guard:** the existing `hopCount >= MAX_HOPS` drop in the relay path fires before `processRouteReport` is reached for any message that has already hit the hop limit. The `data[1] < 10` check is a secondary defence for malformed messages.

**Replay protection:** route report messages pass through the existing `isReplay()` check — no special casing.

## Data Flow

```
Node C (origin)
  → sends MESH_TYPE_ROUTE_REPORT, data = [B3][0][...zeroed...]

Node B (relay)
  → receives, appends own MAC at data[2], sets data[1]=1
  → forwards

Node A (relay)
  → receives, appends own MAC at data[8], sets data[1]=2
  → forwards

Master / Serial adapter
  → receives, passes to server

Server reconstructs:
  originMacAddress  = Node C MAC   (from message header)
  data[2..7]        = Node B MAC   (first relay)
  data[8..13]       = Node A MAC   (second relay)

Full path: C → B → A → master
```

## Out of Scope

- Hub parsing, storage, and API exposure of route reports (separate spec in lattice-hub).
- Changes to functional message types (`MESH_TYPE_ADAPTER_DATA`, health reports, commands).
- Light mode or simulation-mode support for route reporting (follows existing DEV_MODE patterns).
